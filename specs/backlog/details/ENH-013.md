# ENH-013: Extract static scoring data to config/constants.yaml

**Status**: resolved  
**Priority**: P1  
**Phase**: 1  
**Date**: 2026-04-06

## Problem

`src/job_hunter/scoring/engine.py` contained hardcoded static data:
- `SKILL_ALIASES` (24 mappings)
- `NON_TECHNICAL_SKILLS` (13 items)
- `metro_cities` (7 cities + aliases)
- `company_rating_bands` (5 bands)
- `experience_penalty_curve`
- `role_overlap_thresholds`
- `work_mode_scores`

This violates the principle that domain knowledge should be configurable without code changes.

## Solution

1. Create `config/constants.yaml` — all domain knowledge in one place
2. Create `src/job_hunter/config/constants.py` — `Constants` Pydantic model + `load_constants()` with `@lru_cache`
3. Refactor `engine.py` — accept `Constants` as parameter instead of reading module-level data
4. Add scoring weights to `user.yaml` — `scoring.skill_weight`, `role_weight`, etc.

## Files Changed

- Added: `config/constants.yaml`
- Added: `src/job_hunter/config/constants.py`
- Modified: `src/job_hunter/config/__init__.py` — add weights to `ScoringConfig`
- Modified: `src/job_hunter/scoring/engine.py` — accept `Constants` param
- Modified: `src/job_hunter/graph/nodes.py` — load constants in score_jobs_node
- Modified: `config/user.yaml` — add scoring weight fields

## Removed Static Data

- `NON_TECHNICAL_Skills` — filter removed (hurts accuracy)
- `_TITLE_STOP_WORDS` — dead code (never used)
- `_split_concatenated_skills` + `preserved` + `known_skill_starts` — hacks for bad listing data

## Result

- Zero static data in scoring engine code
- All tunables editable via `config/constants.yaml`
- Scoring weights editable via `config/user.yaml`
- Code reduced by ~270 lines

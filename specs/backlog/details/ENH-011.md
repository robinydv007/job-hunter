# Enhancement: Improve search_naukri with more parameters

## Problem

The `search_naukri()` function in `src/job_hunter/search/naukri.py` has limited parameters. As we add more configuration options and anti-blocking features, passing everything through `days_old` and `max_jobs_per_query` is insufficient.

## Proposed Parameters

```python
def search_naukri(
    profile: ResumeProfile,
    search_config: SearchConfig,
    page: Page,
    days_old: int = 7,
    max_jobs_per_query: int = 100,
    experience_filter: int | None = None,      # Filter by years of experience
    salary_min: float | None = None,           # Minimum salary filter
    work_mode: str | None = None,              # remote/hybrid/onsite
    sort_by: str = "relevance",                 # relevance/date/salary
    include_disabled: bool = False,             # Include disabled jobs
    location_preference: str = "preferred",     # preferred/pan-india
) -> list[dict[str, Any]]:
```

## Rationale

- `experience_filter`: Match user's experience against job requirements
- `salary_min`: Filter out jobs below user's minimum salary expectation
- `work_mode`: Filter by remote/hybrid/onsite (ENH-002)
- `sort_by`: Allow sorting by date (freshest first) or relevance
- `include_disabled`: Some jobs are disabled but still searchable
- `location_preference`: Enable Pan-India fallback (ENH-010)

## Implementation Approach

1. Add new parameters to `search_naukri()`
2. Pass through to `scrape_jobs_from_page()` as needed
3. Build query string with additional filters where supported by Naukri
4. Post-filter in Python for filters not supported by Naukri URL params

## Files Changed

- `src/job_hunter/search/naukri.py` — add parameters to `search_naukri()` and `scrape_jobs_from_page()`

## Acceptance Criteria

- All new parameters have sensible defaults — existing behavior unchanged
- Parameters are optional — old code continues to work
- Post-filtering happens after scrape, not before

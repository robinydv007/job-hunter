# Enhancement: Overhaul Search Query Generation

## Problem

### Current query building (`_build_search_queries`)
- Builds Cartesian product of roles × skills × locations → creates overly narrow queries like "Technical Lead Docker"
- Hardcoded cap of 10 unique queries (`naukri.py:53`) — if first 2-3 roles return nothing, remaining roles never get searched
- Skills embedded in search query are redundant — scoring engine already matches skills post-scrape

### Current capping — all hardcoded, inconsistent, or ignored
| Cap | Current value | Where | Issue |
|-----|--------------|-------|-------|
| Roles used | Top 3 | `naukri.py:23` | Not configurable |
| Skills per role | Top 3 tech_stack / 5 skills | `naukri.py:24` | Should be removed from query |
| Locations | Top 3 | `naukri.py:30` | Not configurable |
| Unique queries | 10 | `naukri.py:53` | Arbitrary, causes role starvation |
| Jobs per query | 15 (default) / 20 (from nodes.py) | `naukri.py:385`, `nodes.py:86` | Inconsistent |
| Total jobs per run | 150 | `naukri.py:402` | Hardcoded, ignores `config.search.max_jobs` |
| Scroll iterations | 3 | `naukri.py:359` | Not configurable |

## Solution

### 1. Query building — role only, no skills
```
roles = profile.target_roles[:config.max_roles]
locations = profile.preferred_locations[:config.max_locations]

if locations:
    queries = [{keyword: role, location: loc} for role in roles for loc in locations]
else:
    queries = [{keyword: role, location: ""} for role in roles]

# Pan-India fallback when locations < max_locations → see ENH-010
# NO artificial cap on query count — let it be the natural product
```

### 2. New config fields in `SearchConfig`
```python
max_roles: int = 5          # cap roles from profile
max_locations: int = 3      # cap locations from profile
max_jobs: int = 0           # 0 = no cap on final output, N = return top N scored jobs
```

### 3. Scrape-level soft limit
- **100 raw jobs per query** — prevents scraping thousands for popular queries
- Applied at extraction time in `scrape_jobs_from_page()`
- Not exposed in config — internal safety limit

### 4. No total raw job cap
- Collect all raw jobs from all queries (could be 0 to 1000+)
- Score all of them
- Filter by `shortlist_threshold`
- Sort by score descending

### 5. Output-level cap (optional)
- If `config.search.max_jobs > 0` → return top N shortlisted jobs
- If `config.search.max_jobs == 0` → return all qualified jobs

### 6. Pipeline flow
```
1. Build queries: roles[:max_roles] × locations[:max_locations]  (role-only)
2. Scrape: up to 100 raw jobs per query (soft limit)
3. Collect: all raw jobs from all queries
4. Score: all raw jobs against profile
5. Filter: keep only jobs >= shortlist_threshold
6. Sort: by score descending
7. Output:
   - If max_jobs > 0 → return top N
   - If max_jobs == 0 → return all qualified
```

## Files Changed
- `src/job_hunter/config.py` — add `max_roles`, `max_locations` to `SearchConfig`
- `src/job_hunter/search/naukri.py` — rewrite `_build_search_queries` (role-only, no skills, no query cap), set 100 raw per query soft limit, remove 150 total cap
- `src/job_hunter/graph/nodes.py` — remove hardcoded `days_old` and `max_jobs_per_query`, pass config values
- `src/job_hunter/graph/state.py` — `shortlisted_jobs` now respects `max_jobs` config in `filter_shortlist_node`
- `config/user.yaml` — update template with new fields

## Acceptance Criteria
- Search queries use role only, no skills
- No arbitrary query count cap — all configured roles get coverage
- 100 raw jobs per query soft limit (internal, not configurable)
- No total raw job cap — score everything scraped
- `max_jobs` in config controls final CSV output (0 = all qualified)
- All previously hardcoded values replaced with config or sensible defaults

# ENH-017: Reorder Pipeline - Parse Resume Before Login

## Overview

Currently, the job-hunter pipeline logs into Naukri **before** parsing the resume. This ordering is problematic for future multi-platform support because:

1. Login happens first, but profile data (needed for personalized job searches) isn't available yet
2. Login logic is duplicated if multiple job boards are added
3. No way to platform-select based on profile data

## Current Flow

```
CLI: Start browser ‚Üí Login to Naukri ‚Üí Build workflow ‚Üí Run pipeline
                             ‚Üì
LangGraph: load_config ‚Üí parse_resume ‚Üí search_jobs ‚Üí ...
```

**Problem:** Login occurs before resume parsing, so profile data is unavailable during login.

## Proposed Flow

```
CLI: Start browser (no login) ‚Üí Build workflow ‚Üí Run pipeline
                                       ‚Üì
LangGraph: load_config ‚Üí parse_resume ‚Üí login ‚Üí search_jobs ‚Üí ...
```

**Benefit:** Resume is parsed first, profile is available, then login happens.

## Implementation Plan

### 1. Add Login Node to LangGraph

Create `login_platforms_node()` in `nodes.py` that:

- Takes `config` (platforms list) and `browser_page` (from CLI start)
- Logs into each platform in parallel using `asyncio.gather()`
- Returns `logged_in_platforms` list for tracking

```python
async def login_platforms_node(state: JobHunterState) -> dict:
    """Login to all configured platforms in parallel."""
    platforms = state["config"].search.platforms
    if not platforms:
        return {"logged_in_platforms": []}
    
    page = state["browser_page"]
    results = await asyncio.gather(
        *[login_platform(page, platform) for platform in platforms]
    )
    
    success = [p for p, r in zip(platforms, results) if r]
    return {"logged_in_platforms": success}
```

### 2. Update State Schema

Add field to `JobHunterState` in `state.py`:

```python
logged_in_platforms: list[str] | None  # Platforms successfully logged into
```

### 3. Reorder Workflow Edges

In `workflow.py`, change:

```python
# Old:
workflow.add_edge("parse_resume", "search_jobs")

# New:
workflow.add_edge("parse_resume", "login")
workflow.add_edge("login", "search_jobs")
```

### 4. Update CLI

Remove login from `cli.py:run_pipeline()`:

```python
# Remove these lines:
logged_in = await browser.login_naukri()
if not logged_in:
    console.print("[red]Failed to login to Naukri. Check credentials.[/]")
    await browser.close()
    raise SystemExit(1)
```

**Keep:** Browser lifecycle (start ‚Üí workflow ‚Üí close)

### 5. Login Platform Logic

Create helper in `browser.py` or `nodes.py`:

```python
async def login_platform(page: Page, platform: str) -> bool:
    """Login to a single platform by name."""
    if platform == "naukri":
        return await login_naukri(page)
    elif platform == "linkedin":
        return await login_linkedin(page)  # Future
    else:
        console.print(f"[yellow]Unknown platform: {platform}[/]")
        return False
```

### 6. Credential Flow

Follow existing pattern:

```
config.naukri.email / config.naukri.password (if set in user.yaml)
    ‚Üì (fallback if not set)
os.getenv("NAUKRI_EMAIL") / "NAUKRI_PASSWORD" (from .env)
```

No code changes needed - existing `login_naukri()` already does this.

## Benefits

1. **Profile-first:** Resume parsed before login, profile available for personalized searches
2. **Multi-platform ready:** Login node can handle multiple platforms in parallel
3. **Graceful failure:** Skip failing platforms, continue with others
4. **Platform selection:** Future: can decide which platforms to login to based on profile

## Future Expansion

This structure easily supports adding:

- LinkedIn scraper ‚Üí `login_linkedin()`
- Hirist/Instahyre ‚Üí `login_hirist()`, `login_instahyre()`
- Conditional login ‚Üí skip platforms based on profile data

# ENH-015: LLM-Based Job Scoring

**Status**: resolved  
**Priority**: P1  
**Phase**: 1 (enhancement after MVP)  
**Date**: 2026-04-06  
**Completed**: 2026-04-07

## Problem

The current rule-based scoring engine works well but has limitations:
- Fixed weights may not capture nuanced job-role fit
- Cannot evaluate company culture, role complexity, or growth potential
- User wants to compare LLM-based scoring against rule-based

## Solution

Add optional LLM-based job scoring that can be enabled via config. When enabled, jobs are scored using an LLM instead of the rule-based engine.

### 1. Configuration (`config/user.yaml`)

```yaml
scoring:
  # ... existing weights ...
  llm_scoring:
    enabled: false           # toggle on/off
    batch_size: 10           # jobs per LLM call
    shortlist_threshold: 30  # separate threshold for LLM scoring
    consider_location: false  # consider location in LLM scoring
    consider_work_mode: false  # consider work_mode (remote/hybrid/onsite) in LLM scoring
```

- `consider_location`: When true, LLM considers job location vs candidate's preferred locations
- `consider_work_mode`: When true, LLM considers remote/hybrid/onsite vs candidate's preference

### 2. Config Model (`src/job_hunter/config/__init__.py`)

Add `ScoringLLMConfig` class with fields: enabled, batch_size, shortlist_threshold, consider_location, consider_work_mode. Integrate into `ScoringConfig`.

### 3. New Module: `src/job_hunter/scoring/llm_scorer.py`

**Function**: `score_jobs_with_llm(jobs, profile, config)` → `list[ScoredJob]`

**Batch Logic**:
- Default 10 jobs per batch (configurable)
- Send batch to LLM with profile + batch of jobs

**Job Fields Sent to LLM**:
All Naukri fields EXCEPT: `link`, `posted_date`, `data_source`, `job_board`, `job_url`

**LLM Prompt** includes:
- **User profile**: name, total_experience_years, skills, tech_stack, target_roles, past_roles, industry_domain, company_size_preference
- **Scoring criteria to CONSIDER**:
  - Skills match
  - Role alignment
  - Experience fit
  - Company rating
  - Company review count
  - Industry match
  - Department match
- **IGNORE**: location, work_mode
- **Output format**: JSON with per-job `{job-id, title, company, score (0-100), reasoning}`

### 3.1 Detailed LLM Prompt

The prompt sent to LLM follows this structure:

```
You are a job matching expert. Score each job against the candidate profile.

## Candidate Profile
- Name: {name}
- Total Experience: {experience} years
- Skills: {skills}
- Tech Stack: {tech_stack}
- Target Roles: {target_roles}
- Past Roles: {past_roles}
- Industry Domain: {industry_domain}
- Company Size Preference: {company_sizes}
- Preferred Locations: {preferred_locations}
- Remote Preference: {remote_preference}

## Jobs to Score
{batch_jobs_json}

## Scoring Criteria

### ALWAYS CONSIDER:
1. **Skills Match**: Does the job require technical skills that the candidate possesses?
2. **Role Alignment**: Does the job title align with the candidate's target or past roles?
3. **Experience Fit**: Does the job's experience requirement match the candidate's experience?
4. **Company Rating**: Naukri company rating (if available) - higher is better
5. **Company Reviews**: Number of reviews indicates company size/stability
6. **Industry Match**: Does the job's industry align with candidate's industry domain?
7. **Department Match**: Does the job's department/team align with candidate's background?

### CONDITIONAL (based on user config):
{conditional_criteria}

## Output Format

Return a JSON object with the following structure:
```json
{
  "scores": [
    {
      "job_id": "unique_identifier_from_job",
      "title": "Job Title",
      "company": "Company Name",
      "score": 85,
      "reasoning": "Brief explanation (50-100 words) covering skills, role, experience, company, industry, department factors"
    },
    ...
  ]
}
```

- Score must be an integer from 0-100
- Provide clear reasoning for each score
- Be consistent and fair in evaluation across all jobs in the batch
```

#### Prompt Variables Explained:
- `{name}`: Candidate's name from profile
- `{experience}`: Total years of experience (e.g., 9)
- `{skills}`: Comma-separated list of candidate's skills (from profile.json)
- `{tech_stack}`: Comma-separated list of candidate's tech stack
- `{target_roles}`: Comma-separated list of roles candidate is targeting
- `{past_roles}`: Comma-separated list of roles candidate has held previously
- `{industry_domain}`: Candidate's primary industry (e.g., "Logistics/E-commerce")
- `{company_sizes}`: Candidate's preferred company sizes (e.g., "mid-size, enterprise")
- `{preferred_locations}`: Candidate's preferred job locations (from profile/config)
- `{remote_preference}`: Candidate's work mode preference (remote/hybrid/onsite)
- `{conditional_criteria}`: Dynamic section based on config flags:
  - If `consider_location: true`: Add "8. **Location Match**: Does job location align with candidate's preferred locations?"
  - If `consider_work_mode: true`: Add "9. **Work Mode Match**: Does job's remote/hybrid/onsite align with candidate's preference?"
  - If both false: Use empty string (only core factors considered)
- `{batch_jobs_json}`: JSON array of jobs (10 at a time), each containing: title, company, location, work_mode, experience_required, salary_lpa, description, required_skills, company_rating, review_count, industry, department

### 4. Node Update (`src/job_hunter/graph/nodes.py` → `score_jobs_node`)

```python
def score_jobs_node(state: JobHunterState) -> dict:
    config = state["config"]
    
    if getattr(config.scoring, "llm_scoring", None) and config.scoring.llm_scoring.enabled:
        # Use LLM scoring
        import asyncio
        from job_hunter.scoring.llm_scorer import score_jobs_with_llm
        scored = asyncio.run(score_jobs_with_llm(state["raw_jobs"], state["profile"], config))
    else:
        # Use existing rule-based scoring
        ...
```

### 5. Backward Compatibility

- Default `enabled: false` — existing rule-based scoring runs by default
- If LLM fails, fall back to rule-based
- Same `ScoredJob` output format for downstream nodes

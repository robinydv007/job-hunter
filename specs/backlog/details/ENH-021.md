# ENH-021: Merge Resume Cache into Single File (`profile_cache.json`)

## Problem

Phase 2a introduced two separate auto-generated files for resume data:

- `data/profile.json` — basic profile (name, skills, experience, roles, etc.)
- `data/profile_detailed.yaml` — rich detail (tech_experience years, achievements, challenges, interests)

These were split with the intention of letting users edit `profile_detailed.yaml` manually. In practice this does **not work** — any re-parse (resume change or `--force-parse`) silently overwrites the file, discarding user edits without warning. The two-file approach adds complexity with no real benefit:

- Two loaders (`load_profile`, `load_detailed_profile`)
- Two writers (`json.dump`, `save_detailed_profile`)
- Two cache-validity checks
- Mixed formats (JSON for basic, YAML for detailed)
- `load_profile_with_detailed()` — a thin wrapper that barely adds value

## Proposed Change

Consolidate into a **single file**: `data/profile_cache.json`

- JSON format (consistent with how basic profile is already stored)
- Contains both the basic profile fields and the detailed fields in one document
- Clearly marked as auto-generated — not for manual editing
- `resume_hash.txt` remains separate (it's a side-effect artifact, not profile data)

### New file structure

```json
{
  "name": "Robin Yadav",
  "email": "...",
  "phone": "...",
  "skills": [...],
  "tech_stack": [...],
  "total_experience_years": 9,
  "past_roles": [...],
  "industry_domain": "...",
  "location_preference": "...",
  "preferred_locations": [],
  "salary_expectation": "...",
  "target_roles": [...],
  "education": [...],
  "summary": "...",
  "detailed": {
    "tech_experience": {"Python": 5, "Selenium": 7},
    "achievements": [...],
    "challenges_solved": [...],
    "interests": [...],
    "key_responsibilities": [...]
  }
}
```

## Files to Modify

| File | Change |
|------|--------|
| `src/job_hunter/resume/parser.py` | Replace `profile.json` + `profile_detailed.yaml` save/load with single `profile_cache.json`; remove `save_detailed_profile`, `load_detailed_profile`, `load_profile_with_detailed`; update `parse_resume_full` to write merged file |
| `src/job_hunter/resume/schema.py` | Add optional `detailed: dict` field to `ResumeProfile` (or keep as flat merge — see notes) |
| `src/job_hunter/graph/nodes.py` | Update `parse_resume_node` to use single loader; update return value |
| `src/job_hunter/graph/state.py` | `detailed_profile` state key remains — populated from the merged cache |
| `src/job_hunter/cli.py` | Update `clean` command to remove `profile_cache.json` instead of old filenames |
| `config/user.yaml` / `config/screening.yaml` | No change |

## Notes

- `profile_cache.json` is never manually edited by the user. User corrections live in `config/profile.yaml` (see ENH-022).
- The `detailed` sub-key keeps clear separation between LLM-extracted fields; downstream code that reads `detailed_profile` from state continues to work unchanged.
- `profile.json` and `profile_detailed.yaml` should be deleted from `data/` after migration.

## Acceptance Criteria

- [ ] Single `data/profile_cache.json` written after resume parse
- [ ] Both basic and detailed fields available from one load call
- [ ] No remaining references to `profile.json` or `profile_detailed.yaml` in source code
- [ ] `clean` CLI command removes `profile_cache.json` and `resume_hash.txt`
- [ ] Resume change detection (hash) continues to work unchanged
- [ ] Subsequent runs use cache without LLM call

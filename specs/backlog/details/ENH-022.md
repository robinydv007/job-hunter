# ENH-022: User-Owned Profile Config (`config/profile.yaml`)

## Problem

The LLM does not always extract resume data correctly:

- Experience years may be wrong (e.g., LLM says 5, user has 9)
- Skills may be incomplete (tools not mentioned by name in the resume)
- Tech-specific years-of-experience may be inaccurate

Additionally, there is information that is never in the resume but is highly valuable for scoring, auto-apply, and LLM question-answering:

- Career goal / why looking for a change (extended version)
- Key strengths the user wants to emphasise
- Preferred company type (product vs service, startup vs enterprise)
- Skills the user is happy to be assessed on even if not in resume

Right now there is **no stable place** for the user to store this information. `profile_detailed.yaml` was supposed to serve this role but gets silently overwritten on every re-parse (see ENH-021).

## Proposed Change

Introduce `config/profile.yaml` — a **user-owned, never auto-overwritten** config file.

It has two sections:

### `overrides` — correct LLM extraction errors

Values here take precedence over `profile_cache.json` at runtime:

```yaml
overrides:
  total_experience_years: 9          # LLM extracted 5, correct to 9
  skills:                            # Append to LLM-extracted skills
    - Performance Testing
    - JIRA
    - BDD/Cucumber
  tech_experience:                   # Correct years per technology
    Selenium: 7                      # LLM said 5
    Python: 4
```

### `enrichment` — information not in the resume

Used by the LLM for question answering, scoring context, and personalisation:

```yaml
enrichment:
  career_goal: "Move into QA Lead or SDET architect roles within 2 years"
  strengths: |
    Strong analytical mindset, fast learner, automation-first thinker.
    Built frameworks from scratch at two organisations.
  what_can_you_bring: |
    9 years of QA leadership, ability to architect test frameworks from scratch,
    and proven track record of cutting regression cycle by 50%.
  reason_for_change: |
    Looking for a role with greater technical ownership and the opportunity
    to lead a QA function end-to-end.
  preferred_company_types:
    - product
    - SaaS
  open_to_contract: false
  additional_skills:
    - "Team mentorship"
    - "Test strategy definition"
```

## Merge Logic at Runtime

After loading `profile_cache.json`, the pipeline merges `config/profile.yaml` overrides:

```
effective_profile = profile_cache.json
                  + config/profile.yaml (overrides win, enrichment appended)
```

The merged result is passed into state as `profile` and `detailed_profile`. The cache file itself is **never modified** by this merge.

## `config/profile.yaml` Template (generated by `job-hunter init`)

```yaml
# config/profile.yaml
# User-owned. Never overwritten by the resume parser.
# Use this to correct LLM extraction errors and add information not in your resume.

overrides:
  # Uncomment and set to correct LLM extraction mistakes
  # total_experience_years: 9
  # skills:
  #   - Performance Testing
  #   - JIRA
  # tech_experience:
  #   Selenium: 7

enrichment:
  # career_goal: ""
  # strengths: ""
  # what_can_you_bring: ""
  # reason_for_change: ""
  # preferred_company_types: []
  # open_to_contract: false
  # additional_skills: []
```

## Relationship to Other Files

```
config/user.yaml      → Search & scoring preferences (what jobs to find)
config/screening.yaml → Application Q&A answers (how to answer forms)
config/profile.yaml   → Who you are (user-editable, stable across re-parses)  ← NEW
data/profile_cache.json → LLM output (auto-generated, never manually edited)
```

## Acceptance Criteria

- [ ] `config/profile.yaml` created with commented template on `job-hunter init`
- [ ] `load_config()` loads and validates `profile.yaml` if it exists (optional — no error if absent)
- [ ] `overrides.total_experience_years` replaces value from cache when set
- [ ] `overrides.skills` are appended (not replaced) to LLM-extracted skills
- [ ] `overrides.tech_experience` values replace per-technology years in detailed profile
- [ ] `enrichment` fields are passed to `detailed_profile` in state for LLM context
- [ ] Re-parsing resume does not modify or delete `config/profile.yaml`
- [ ] Pipeline works unchanged when `config/profile.yaml` is absent
